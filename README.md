# ClashFox

ClashFox 是一个免构建的 Firefox WebExtension，用来读取 mihomo 外部控制器的 `/connections`，并在工具栏弹窗中显示当前标签页域名、当前完整地址、全部连接，以及当前页面资源可能对应到的 mihomo 连接。

0.2.3 开始，弹窗和扩展图标统一为简洁卡通狐风格，并支持严格、智能、宽松三档匹配策略。

## 使用

1. 在 mihomo 配置里启用外部控制器：

   ```yaml
   external-controller: 127.0.0.1:9090
   secret: ""
   ```

   远程控制器示例：

   ```yaml
   external-controller: 192.168.20.2:9090
   secret: ""
   ```

2. 打开 Firefox 的 `about:debugging#/runtime/this-firefox`。
3. 点击「临时载入附加组件」，选择本仓库里的 `manifest.json`。
4. 打开扩展设置，填写 API 控制器地址和 `secret`。例如 yacd 链接里出现 `hostname=192.168.20.2&port=9090&secret=12210127` 时，这里填写：

   - 控制器地址：`http://192.168.20.2:9090`
   - Secret：``

## 当前能力

- 默认连接 `http://127.0.0.1:9090/connections`，也支持设置为局域网或远程控制器地址。
- 工具栏 badge 显示当前过滤模式下匹配到的 mihomo 连接数量。
- 弹窗支持快速切换多个 mihomo 后端，切换后会立即刷新当前标签页连接。
- 弹窗支持快速切换四种过滤模式：当前主域名、当前完整地址、页面资源、全部连接。
- 弹窗和设置页支持浅色/深色主题切换，整体视觉使用统一的卡通狐风格。
- 弹窗显示匹配连接的目标地址、连接类型、规则、代理链路、上下行流量。
- 设置页支持配置多个后端、控制器地址、`secret`、刷新间隔和匹配策略。
- 请求 mihomo API 时会携带 `Authorization: Bearer <secret>`。
- 当前标签页不是 HTTP/HTTPS 页面时，会自动按「全部连接」模式请求 mihomo，避免在 `about:`、扩展页等页面误判为后端不可用。

## 多后端

在扩展设置页可以新增、删除、测试多个后端。每个后端包含：

- 后端名称
- API 控制器地址，例如 `http://192.168.20.2:9090`
- Secret

保存后，弹窗里的「后端」下拉菜单会显示这些后端。选择不同后端时，ClashFox 会保存当前选择并立刻重新请求 `/connections`。

## 过滤模式

弹窗中可以快速切换四种过滤模式：

- 主域名：按当前页面主域名展示连接，例如 `www.example.com` 会匹配 `example.com` 及其子域名。
- 完整地址：按当前页面完整主机地址精确展示连接，例如只匹配 `www.example.com`。
- 页面资源：记录当前页面 URL 生命周期内捕获到的 HTTP/HTTPS/WebSocket 资源请求，并尽量关联这些资源可能对应到的 mihomo 连接。统计卡片和 badge 只统计这些资源可能对应到的 mihomo 连接，不再把资源条数作为主要统计口径。
- 全部：不按当前页面过滤，展示当前后端 `/connections` 返回的所有连接。

mihomo 连接数据通常不包含 URL path 或 query，所以「完整地址」按主机名和显式端口匹配，不按 `/path?query` 匹配。

## 匹配说明

mihomo 的连接数据不包含 Firefox 标签页 ID，所以 ClashFox 以当前标签页的 hostname 匹配连接里的 `metadata.host`、`metadata.remoteDestination`、`metadata.destinationIP` 等字段。对于只暴露 IP、未保留域名，或页面里的第三方子资源连接，可能无法完全归属到当前标签页。

页面资源模式也只能做近似关联：浏览器请求是 URL 级别，mihomo `/connections` 是 TCP/UDP 连接级别。HTTP/2、HTTP/3、WebSocket、连接复用、DNS/IP 映射变化都会导致二者无法 100% 一一对应。因此页面资源模式仅用于辅助判断“这些页面资源可能经过了哪些 mihomo 连接”，不能作为严格的 URL 级流量统计。

匹配策略有三档：严格模式只接受精确 Host/IP，智能模式以精确 Host/IP/端口为优先并使用主域名辅助判断，宽松模式允许更多同域名后缀归属。旧版“匹配同域名后缀”配置会自动迁移到对应策略。

## 权限说明

为了让控制器地址可以配置为 `127.0.0.1`、局域网 IP、域名或 HTTPS 地址，扩展在 `manifest.json` 中申请了 `http://*/*` 与 `https://*/*` 请求权限。

为了记录页面发起的 WebSocket 握手请求，扩展也申请了 `ws://*/*` 与 `wss://*/*` 权限，并在后台 `webRequest` 过滤器中同时监听 HTTP、HTTPS、WS、WSS。

## 页面资源监控

0.2.1 起调整「页面资源」模式：

- 支持捕获 `http://`、`https://`、`ws://`、`wss://` 请求。
- 后台在主文档请求开始时创建新的页面 URL 生命周期；当前标签页 URL 变化时会清空旧资源。
- 弹窗只展示当前页面 URL 生命周期内捕获到的资源，避免旧页面资源残留到新页面。
- 顶部统计和 badge 只按资源可能对应到的 mihomo 连接计算，不再统计资源条数、资源域名数。
- 仍然会在资源列表中显示请求类型、方法、状态码、耗时、服务端 IP、缓存命中、第三方资源标记，以及尽量关联到的规则、代理链路和流量。

注意：浏览器 `webRequest` 只能捕获扩展监听之后发生的请求。打开弹窗前已经完成的请求可能不会出现；如果页面资源列表为空，请刷新当前页面后再查看。
